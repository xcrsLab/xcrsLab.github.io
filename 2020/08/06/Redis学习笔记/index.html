        Redis介绍

        概述

        Redis(Remote Dictionary Server),远程字典服务,是一个开源的使用ANSI C语言编写、支持网络、可基于内存亦可持久化的日志型、Key-Value数据库，并提供多种语言的API
        Redis支持主从同步。数据可以从主服务器向任意数量的从服务器上同步，从服务器可以是关联其他从服务器的主服务器。这使得Redis可执行单层树复制。存盘可以有意无意的对数据进行写操作。由于完全实现了发布/订阅机制，使得从数据库在任何地方同步树时，可订阅一个频道并接收主服务器完整的消息发布记录。同步对读取操作的可扩展性和数据冗余很有帮助。

        性能

        下面是官方的bench-mark数据：
        测试完成了50个并发执行100000个请求。
        设置和获取的值是一个256字节字符串。
        Linux box是运行Linux 2.6,这是X3320 Xeon 2.5 ghz。
        文本执行使用loopback接口(127.0.0.1)。
        结果:读的速度是110000次/s,写的速度是81000次/s 。

        作用

      1. 内存存储，持久化，内存中是断电即失，redis可持久化（rdb,aof）
      2. 效率高，可做告诉缓存
      3. 发布订阅系统
      4. 地图信息分析
      5. 计时器计数器

特征

  1. 多样的数据类型
  2. 持久化
  3. 集群
  4. 事务

单线程

Redis基于内存操作，cpu不是性能瓶颈，是根据机器的内存和网络带宽
Redis是C语言写的，10万+的QPS
为什么单线程还这么快？
误区1：高性能服务器一定是多线程的
误区2：多线程一定比单线程效率高
Redis是将所有的数据全部存放在内存中的，用单线程去操作效率就是最高的，多线程（CPU会进行上下文的切换，是耗时的操作），对于内存系统来说，没有上下文切换就是效率最高的

事务

特性

Redis事务的本质：一组命令的集合，没有隔离级别的概念

  1. 一次性：一组命令被序列比，一次执行
  2. 顺序性：序列化后按顺序执行
  3. 排他性
    Redis单条命令是有原子性的，但是事务不能保证原子性

    过程

  4. 开启事务（multi）
  5. 命令入队（……）
  6. 执行事务（exec）

放弃事务

  1. 编译时异常：代码错误，命令都不会执行
  2. 运行时异常：如果事务中存在语法性，那么执行命令的时候，其他命令可正常执行

aof配置

Redis持久化

redis是内存数据库，如果不将内存中的数据保存到磁盘，一但服务器进程退出，数据也会消失，所以redis提供了持久化功能

RDB(Redis DataBase)

在指定的时间间隔内，将内存的数据写入磁盘（快照SnapShot），恢复时，就将快照中的文件读取到内存

文件名称


rdb保存的文件名称是dump.rdb

保存频率

60s内修改了5次，则进行保存

触发机制

  1. save的规则满足的情况下，会自动触发rdb规则
  2. 执行flushall命令，也能触发rdb规则
  3. 退出redis也会生成edb文件

备份后就会自动生成dump.rdb文件

恢复rdb文件

将rdb文件放在redis的启动目录，启动后会自动回复rdb的数据

优缺点

优点：

  1. 适合大规模数据的恢复
  2. 对数据的完整性要求不高

缺点：

  1. 需要一定的时间间隔，如果宕机了，最后一个时间间隔的数据会丢失
  2. fork进程的时候，会占用一定的内存空间

AOF（Append Only File）

记录每次的写入和修改操作，启动时，将这些操作同步到库中

文件名称

redis保存的是append only file.aof文件，默认不开启
如果aof损坏，redis是无法启动的，可以使用redis-check.aof –fix进行修复

重写规则


如果aof文件大于64m，fork一个新的进程将文件重写，aof默认的是文件无限制的追加，会使文件越来越大

优缺点

优点：

  1. 每次修改都需要同步，文件完整性梗好
  2. 每秒同步一次，可能丢失一秒的数据
  3. 从不同步，效率最高

缺点：
4. 相对于数据文件，aof要远大于rdb文件，数据恢复速度慢
5. aof运行效率比rdb要低，所以默认使用rdb

Redis主从复制

概念

主从复制：将一台redis服务器（master）的数据，复制到其他redis服务器（slave）

作用

  1. 数据冗余：实现了热备份，是另一种持久化方式
  2. 故障恢复：主节点故障时，可以由从节点提供服务
  3. 负载均衡：实现读写分离，主节点负责写服务，从节点提供读服务，尤其时写少读多时，可以配置多个从节点用于读服务
  4. 高可用性：是哨兵和集群实施的基础

读写分离

主机可以写，从机只能读，主机的所有信息会被从机保存，主机宕机后，从机可以读取数据，但无法写入

复制原理

slave启动成功后，连接到master会发送一个sync同步命令
master接到命令后，启动后台的存盘进程，收集到用于修改数据的命令集合，将该文件传输到slave完成一次同步
全量复制：slave接收到数据，将其存盘并加载到内存中
增量复制：master继续将修改的命令依次传到slave，完成同步
重新连接时，会进行一次全量复制

哨兵模式

哨兵是一个独立的进程，监控多个redis实例的运行状况

作用

  1. 通过发送命令，让redis返回运行状态进行监控，包括主机与从机

  2. 当哨兵发现主机宕机后，会自动将slave切换为master，然后通过发布订阅模式通知其他从服务修改配置文件，切换主机

    然而哨兵本身会出现宕机的情况，可以使用多个哨兵互相监控，形成多哨兵模式
    如果主机宕机，哨兵1先检测到这个结果，不会马上切换主机，只是哨兵1认为主机不可用，这种现象称为主观下线，当后面的主机也检测到主机不可用，并达到一定的数量时，哨兵之间会进行一次投票，切换主机成功后，通过发布订阅模式，使其他从机切换到新的主机，这种现象称为客观下线，如果主机重新上线，会变成新主机的从机

配置

配置哨兵的配置文件sentinel.conf

优缺点

优点：

  1. 支持哨兵集群，保证高可用性
  2. 主从可以自动切换，保证高可用性

缺点：

  1. redis不好在线扩容
  2. 配置繁琐

缓存穿透

概念

用户发起查询请求，首先在缓存中进行查询，缓存中查询不到，再去查询数据库，如果数据库仍然查询不到并发起了多次查询，在未命中缓存的情况下，无法减轻数据库的压力，称为缓存穿透

解决方案

布隆过滤器

是一种hash形式的数据结构，将所有的可能查询到的参数以hash形式存储，先在控制进行校验，不符合则放弃查询

缓存空对象

当缓存未命中，且数据库中也不存在时，将返回的空对象也缓存起来，设置一个过期时间，在该段时间内，可以保证缓存被命中
但是存在两个问题：

  1. 空对象会占用 一定空间，存储更多的键值
  2. 即使设计过期时间，还是会存在一段缓存和数据库一定时间的不一致，无法保证一致性

缓存击穿

概念

一个访问量非常高的key值，被大量集中访问，当这个key消失时，大量的请求访问数据库

解决方案

设置热点数据不过期

不会产生击穿问题

加互斥锁

分布式锁，保证每个key只有一个线程去访问，其他的线程需要等待

缓存雪崩

概念

  1. 指某个时间段内，缓存集中失效，导致大量的请求直接访问数据库。
  2. redis宕机

解决方案

redis集群

保证高可用性

限流降级

在缓存失效后，通过加锁或队列的方式限制访问的并发量

数据预热

正式部署前，先将数据访问一遍，加载到内存，在大并发访问前，手动触发访问，使失效的时间尽量均匀
